> **Lý do quan trọng — Không được fit lại preprocessor:**  
> Preprocessor phải là bản đã fit từ tập train, tuyệt đối không được fit lại trên dữ liệu mới. Nếu fit lại, các thông số chuẩn hóa (mean, std, encoder categories) sẽ thay đổi → model nhận input không khớp với lúc huấn luyện → kết quả sai mà **không có lỗi nào xuất hiện** (gọi là training-serving skew).

**Script thực hiện:** `scripts/register_pipeline.py`

| Bước | Hành động |
|------|-----------|
| 1 | Load preprocessor từ `artifacts/` bằng `load_joblib_artifact()` |
| 2 | Load model champion từ `artifacts/final_model.joblib` |
| 3 | Đóng gói thành sklearn Pipeline 3 bước bằng `build_full_sklearn_pipeline()` |
| 4 | Log Pipeline lên MLflow bằng `mlflow.sklearn.log_model()` |
| 5 | Gán alias `@champion` cho phiên bản vừa đăng ký bằng `client.set_registered_model_alias()` |

### 🏗️ PHASE 1 — Xây Dựng Sklearn Pipeline

**Mục đích:**
Tạo một pipeline xử lý khép kín: nhận dữ liệu thô → xử lý feature → dự đoán.

**File chính:** `scripts/inference_pipeline.py`

#### 1.1 — FeatureBuilder (Bước 1 của Pipeline)

Đây là transformer chịu trách nhiệm biến dữ liệu thô thành 28 features chuẩn cho mô hình.

| Hành động | Hàm thực hiện |
|-----------|--------------|
| Chặn data leakage — từ chối nếu có cột cấm | Kiểm tra `BANNED_FEATURES` trong `FeatureBuilder.transform()` |
| Tạo flag đa nhãn cho danh mục sản phẩm | Multi-hot encoding category/segment/size |
| Gom nhiều dòng về 1 đơn hàng (grain order_id) | `groupby("order_id").agg()` |
| Tính số ngày khách hàng gắn bó | `customer_tenure_days` = order_date − signup_date |
| Phân nhóm thâm niên → 4 bucket | `pd.cut()` với `TENURE_BINS` cố định |
| Tính log giá trị thanh toán | `log_payment_value` = `log1p(payment_value)` |
| Tính tỷ lệ giảm giá | `discount_ratio` = discount / gross_value |
| Tạo flag COD và có giảm giá | `is_cod`, `is_discounted` |

**28 Features đầu ra:**

| Nhóm | Số lượng | Ví dụ |
|------|----------|-------|
| Numeric | 5 | `customer_tenure_days`, `total_quantity`, `discount_ratio`, `log_payment_value`, `unique_product_count` |
| Categorical | 6 | `payment_method`, `device_type`, `order_source`, `tenure_group`, `age_group`, `gender` |
| Binary (multi-hot) | 16 | `category_Casual`, `segment_Everyday`, `size_M`, `is_cod`, `is_discounted`, ... |
| Quantile | 1 | `payment_value` |

**Features bị cấm (data leakage):**

| Feature | Lý do cấm |
|---------|-----------|
| `order_status` | Chính là nhãn cần dự đoán |
| `mean_product_return_rate` | Lấy từ file `returns.csv` — thông tin tương lai |
| `max_product_return_rate` | Tương tự |
| `high_risk_product_count` | Tương tự |
| `has_return_record` | Tương tự |

#### 1.2 — ColumnTransformer (Bước 2 của Pipeline)

Load bộ tiền xử lý đã fit từ `artifacts/preprocessor_v1_outer_train.joblib`. **Chỉ gọi `.transform()`, không fit lại.**

Xử lý đồng thời 4 nhóm cột:
- **Numeric (5 cột):** Impute bằng median → StandardScaler
- **Categorical (6 cột):** Impute bằng mode → OneHotEncoder
- **Binary (16 cột):** Impute bằng 0 → giữ nguyên
- **Quantile (1 cột):** Impute bằng median → StandardScaler

#### 1.3 — ThresholdedClassifierWrapper (Bước 3 của Pipeline)

Bọc model từ MLflow, tích hợp ngưỡng quyết định cố định vào trong:
- `predict_proba()` → xác suất [P(không return), P(return)]
- `predict()` → áp ngưỡng **0.063357** (được tính là ngưỡng tốt cho model nhất) → nhãn 0/1

> **Tại sao ngưỡng thấp (0.063357)?**  
> Trong bài toán dự đoán hoàn hàng, bỏ sót một đơn có nguy cơ hoàn hàng thường tốn kém hơn cảnh báo nhầm. Ngưỡng thấp tăng Recall — phát hiện nhiều trường hợp nguy cơ hơn.

#### 1.4 — Hàm hỗ trợ tương thích phiên bản cũ

**Vấn đề:** `SimpleImputer` từ sklearn phiên bản cũ bị thiếu thuộc tính `_fill_dtype` khi load từ file `.joblib`.

**Giải pháp:** Hàm `load_joblib_artifact()` tự động duyệt đệ quy artifact sau khi load, vá thuộc tính còn thiếu bằng `_fix_all_imputers_in_artifact()`, đảm bảo không crash.

**Kết quả Phase 1:**
- ✅ Pipeline 3 bước hoạt động ổn định
- ✅ Phát hiện và chặn data leakage tự động
- ✅ Tương thích với preprocessor từ các phiên bản sklearn cũ
- ✅ Ngưỡng quyết định được lưu trong model, không hardcode ngoài

### 🌐 PHASE 2 — Xây Dựng FastAPI

**Mục đích:**
Bọc pipeline thành API HTTP để bất kỳ hệ thống nào có thể gửi dữ liệu đơn hàng và nhận kết quả dự đoán ngay lập tức.

**File chính:** `api/app.py` — **FastAPI version 6.0**

#### 2.1 — Cơ chế khởi động (Lifespan Pattern)

Khi server bắt đầu chạy, pipeline được **load một lần duy nhất** từ MLflow vào bộ nhớ qua `_load_pipeline()`. Mọi request sau đó dùng lại pipeline này mà không load lại.

> Lý do: Load model mất vài giây — nếu load lại cho mỗi request thì API không thể đáp ứng được.

Nếu load thất bại, server vẫn khởi động bình thường nhưng `/health` sẽ báo `pipeline_loaded: false`.

#### 2.2 — Cấu hình qua biến môi trường

| Biến | Mặc định | Ý nghĩa |
|------|----------|---------|
| `MLFLOW_TRACKING_URI` | `http://localhost:5000` | Địa chỉ MLflow server |
| `MLFLOW_MODEL_URI` | `models:/customer-return-champion@champion` | URI model trong Registry |

#### 2.3 — Các Endpoints

**`GET /health`** — Kiểm tra trạng thái hệ thống

Trả về: trạng thái pipeline, địa chỉ MLflow, ngưỡng hiện tại.

```
Kết quả mẫu: {"status": "ok", "pipeline_loaded": true, "threshold": 0.063357}
```

**`POST /predict`** — Endpoint dự đoán chính

Hỗ trợ 2 dạng input:

| `input_type` | Mô tả | Dùng khi |
|--------------|-------|----------|
| `batch` | Danh sách nhiều đơn hàng | Dự đoán hàng loạt (ví dụ : bảng dữ liệu) |
| `single_record` | Đúng 1 đơn hàng | Dự đoán cho real-time  |

Quá trình xử lý trong `/predict`:
1. `_prepare_master_df()` — chuyển JSON → DataFrame
2. Kiểm tra có cột `order_id` không
3. `_run_pipeline()` — đưa qua Pipeline → lấy nhãn + xác suất + ngưỡng
4. Đóng gói kết quả thành `PredictResponse`

Kết quả trả về cho mỗi đơn hàng:

| Trường | Ý nghĩa |
|--------|---------|
| `order_id` | Mã đơn hàng |
| `return_probability` | Xác suất đơn bị hoàn (0.0 → 1.0) |
| `prediction` | Nhãn dự đoán (0 = không hoàn, 1 = có nguy cơ hoàn) |
| `threshold` | Ngưỡng đang dùng (lấy từ model, không hardcode) |

**Kết quả Phase 2:**
- ✅ API chạy ổn định trên cổng 8000
- ✅ Swagger UI tại `http://localhost:8000/docs` để test thủ công

## 5. Đánh Giá Tổng Thể Luồng MLOps (Góc Nhìn Chuyên Gia)

> *Dưới đây là đánh giá khách quan từ góc nhìn deployment — những điểm tốt cần giữ, những điểm cần cải thiện trong thực tế.*

### ✅ Những điểm làm đúng — cần giữ nguyên

| Điểm mạnh | Lý do quan trọng |
|------------|-------------------|
| **Singleton pattern khi load model** | Tự nhiên — đúng cách. Load 1 lần khi startup, tái dùng cho mọi request. Nếu không làm vậy, mỗi request mất 5–10 giây chỉ để load model — API không thể dùng được |
| **Leakage gate trong FeatureBuilder** | Rất tốt. Chặn data leakage ngay tại runtime, không chỉ lúc train. Ðây là tư duy đúng của người hiểu và tôn trọng deployment |
| **MLflow Registry + alias @champion** | Sử dụng đúng pattern quản lý model. Alias tách rời version số — có thể promote model mới lên @champion mà không đổi URI |
| **Dockerize API rửi push Docker Hub** | Portable — chạy được trên mọi máy có Docker. Nền tảng tốt để deploy lên cloud sau này |
| **Threshold tích hợp vào Pipeline** | Dễ audit — ai load model nào thì dùng đúng threshold của model đó, không bị lệch |

---

### ⚠️ Những điểm còn thiếu — đánh giá theo luồng MLOps chuẩn

#### 🔴 Điểm nghiếm trọng: MLflow chưa có lớp lưu trữ tập trung

Toàn bộ MLflow — tracking URI, artifacts, model registry — đang nằm trên máy cá nhân.

Hệ quả thực tế:
- **Không ai khác xem được** experiments hay metrics khi cần review
- **Artifacts của MLflow (model files) lưu local** — nếu ổ cứng hỏng, mất luôn mọi model version đã đăng ký
- **API trong Docker kết nối về `localhost`** — địa chỉ này trong container trỏ vào chính container, không phải máy host → phải workaround bằng `host.docker.internal` (không chạy trên Linux)
- Đây là **anti-pattern nghiêm trọng** trong MLOps: MLflow phải là shared service, không phải local tool

> **Chuẩn công nghiệp:** MLflow server chạy trên máy chủ riêng, artifacts lưu trên S3/GCS/Azure Blob, database backend là PostgreSQL — hoàn toàn tách biệt với máy dev

---

#### 🟡 Điểm cần cải thiện: Không có validation schema ở tầng API

Hiện tại API chỉ kiểm tra `order_id` có tồn tại không. Chưa có:
- Kiểm tra **kiểu dữ liệu** của từng field (ví dụ: `age_group` phải là string, `quantity` phải là số dương)
- Kiểm tra **giá trị hợp lệ** (ví dụ: `gender` chỉ nhận "Male"/"Female", không nhận giá trị lạ)
- **Cảnh báo** khi giá trị nằm xa phân phối lúc train (ví dụ: `payment_value` = 999,000,000 — bất thường)

Hệ quả: Pipeline có thể nhận dữ liệu sai, chạy không lỗi nhưng **trả về kết quả vô nghĩa** mà không ai biết.

---

#### 🟡 Điểm cần cải thiện: Không theo dõi được mô hình sau khi triển khai

Sau khi model được deploy, không có cơ chế nào phát hiện:
- **Prediction drift:** Tỷ lệ dự đoán `return=1` đột ngột tăng/giảm bất thường
- **Data drift:** Phân phối input đầu vào thay đổi so với lúc train
- **Latency degradation:** API bắt đầu chậm dần theo thời gian

Trong thực tế đây là lý do mô hình "chạy đúng" lúc deploy nhưng **cho kết quả xấu sau vài tuần**.

---

#### 🟡 Điểm cần cải thiện: Chưa có cơ chế cập nhật model không downtime

Nếu muốn cập nhật champion model mới, bắt buộc phải restart API server → có khoảng thời gian API không phục vụ được. Trong môi trường production điều này không chấp nhận được.

---

#### 🟢 Điểm bổ sung: API chưa có xác thực và giới hạn truy cập

Bất kỳ ai biết địa chỉ API đều có thể gọi `/predict` không giới hạn. Ở giai đoạn nội bộ (intern/datathon) điều này có thể chấp nhận, nhưng cần ghi nhớ để bổ sung trước khi mở rộng.

## 6. Kế Hoạch Tiếp Theo

Đề xuất theo thứ tự ưu tiên dựa trên mức độ ảnh hưởng đến luồng MLOps:

### 🔴 Ưu tiên 1 — Giải quyết ngay (Nền tảng MLOps)

| Việc cần làm | Lý do |
|-------------|-------|
| **Deploy MLflow lên server cố định** (VD: một máy chủ nội bộ, hoặc EC2/Render/Railway miễn phí) | Đây là điều kiện cần để mọi thứ khác hoạt động đúng nghĩa MLOps |
| **Cấu hình artifact store** của MLflow trỏ vào một thư mục shared hoặc S3 bucket | Tránh mất model khi máy hỏng, ai cũng download được artifacts |
| **Cập nhật `MLFLOW_TRACKING_URI`** trong `docker-compose.yml` trỏ đến server cố định thay vì `localhost` | Giải quyết triệt để vấn đề container ↔ MLflow |

### 🟡 Ưu tiên 2 — Cải thiện độ tin cậy

| Việc cần làm | Lý do |
|-------------|-------|
| **Thêm input schema validation** vào `/predict` — kiểm tra kiểu dữ liệu và dải giá trị hợp lệ cho từng field | Phát hiện sớm dữ liệu sai trước khi vào pipeline |
| **Thêm endpoint `/reload-model`** để hot-reload champion model không cần restart server | Cập nhật model mới mà không downtime |
| **Log mỗi request**: số bản ghi, `positive_rate`, latency xử lý, timestamp → ghi vào file hoặc MLflow | Nền tảng để phát hiện drift về sau |

### 🟢 Ưu tiên 3 — Hoàn thiện dài hạn

| Việc cần làm | Lý do |
|-------------|-------|
| **Theo dõi `positive_rate` theo thời gian** — cảnh báo nếu lệch quá ngưỡng so với baseline lúc train | Phát hiện prediction drift sớm |
| **Tách threshold ra ngoài Pipeline** — đọc từ biến môi trường hoặc config file | Điều chỉnh ngưỡng không cần rebuild Pipeline |
| **Thêm xác thực API** (API key hoặc token đơn giản) trước khi mở rộng phạm vi sử dụng | Kiểm soát ai được gọi API |
